十分钟搭建个人博客(Hexo+GithubPage+AWS Route53) [05.2020]

疫情期间保持学习节奏，决定利用博客来记录和总结学习的点滴进度

在网上看到了Hexo是个很不错的静态页面生成框架，之前浏览其他大佬的页面时候也看到过这个名字，想拿来试试

但是鉴于能Google到的教程是在太古老，导致写Blog5分钟，部署搭环境两小时

决定用个人博客的第一篇就来share，在时至今日[2020年 5月]用Hexo搭建博客中遇到的坑与注意事项

希望能够帮到看到这篇文章的你


大致流程：

Hexo环境配置 -> Hexo本地创建静态页面 -> 本地起服务测试
链接Github仓库，利用Github Page测试线上页面
个性化域名连接(以AWS Route 53为例)



1. Hexo前提环境配置

有一点npm使用经历或基础的同学应该已经安装好了 可以略过到2

(1) 安装Git


Mac: 使用 Homebrew 或 MacPorts 比较方便
Windows: 下载并安装 git
Linux (Ubuntu, Debian)：sudo apt-get install git-core
Linux (Fedora, Red Hat, CentOS)：sudo yum install git-core

(2) 安装Node.js


Mac：依然是 Homebrew 或 MacPorts 方便些。
Windows：通过 nvs（推荐）或者nvm 安装。
Linux（DEB/RPM-based）：从 NodeSource 安装。

2. Hexo安装

还不够熟悉npm操作的同学可以全局安装

有可能会permission deny, 请在命令前加sudo 然后输入账户密码


足够熟悉的同学可以局部安装hexo

别忘了先npm init

3. Github创建仓库与Github Page

如果还没有Github账户的同学请先 注册,用户名尽量全部小写

(1) 新建项目

！注意：项目名必须以你的用户名.github.io命名
如:username.github.io
原因是因为Github page生成网页会根据你的repo项目命名来生成，如果不是用户名.github.io的格式url格式会变 从而导致链接你的自定义域名失败

(2) Github Page

通过创建初始readme.md或者后面push上去初始的hexo文件后
点击仓库的Setting页面，向下拉会有GitHub Pages一栏。[ 明天补图 ]
下方会有刚才的用户名.github.io显示，稍等一会儿即可点击页面看到GithubPage初始页面 [ 明天补图 ]

3. Hexo创建本地项目

推荐在一个新的干净文件夹下存放Hexo生成的本地项目文件和博客目录比如’/Users/xxx/Projects/HexoBlog’
在之前步骤安装成功的标志是hexo -v或者npx hexo -v能看到Hexo所需相关依赖的版本号

确认安装成功后用hexo init在一个空文件夹下创建初始文件
然后使用npm i安装npm的依赖包 (依然会有权限问题 老样子sudo解决)

然后使用hexo g/npx hexo g生成静态页面
注：这个命令每次要生成新post后都会需要
再用hexo s/npx hexo s启动本地服务
打开 http://localhost:4000/ 就可以看到初始博客效果了 [ 明天补图 ]

No.1 of Top 10 Java Design Pattern

第一篇文章分享Java里 个人认为最常用、最重要的十个设计模式中的第一个 单例

1. Singleton

Level：Hard

Hard 原因很简单，看似最简单最基础的设计模式却拥有最多的变种

Singleton Pattern

restricts the instantiation of a class and ensures that only one instance of the class exists in the Java virtual machine.

My thought:

    1. make only one instance in jvm -> static constructor
    1. also need to restrict instantiation from other
  • -> private constructor
    1. provide a global access point to get the instance
  • -> public getMethod (private variable based on encapsulation principle)
    1. final? The final modifier allows the Java compiler and runtime
  • to make good optimization and thread-safety decisions.

1. Eager initialization

2. Static block initialization

avoid create instance more than one time and good for error handle

Eager and static block both create the instance of singleton

even if the class is not being used so far (not lazy initialization)

3. Lazy Initialization

create instance when we call get method

4. Thread safe Singleton

only with ‘synchronized’ key word, only one thread can execute this method at a time.

5. Reflection will destroy singleton

since it create a new instance by the reflect constructor

6. The best practice of singleton in Java is enum

which has all the features we need,
like instantiated only once, globally accessible
but drawback is not support lazy load

7. Serialized Singleton

sometime, in distributed systems, we need implement serializable interface in singleton
so we could store its state in file system and future retrieve
